# MWAgent 자동 업데이트(run.agent.bat) 권한 상승 파훼 및 해결 과정

## 1. 개요
* **날짜:** 2026-03-11
* **작업명:** 망분리 환경 MWAgent 자동 업데이트(restart.bat) 실패에 따른 권한 문제 분석 및 해결
* **증상:** 서버를 통해 MWAgent에 업데이트 패키지 전송 및 `run.agent.bat` 실행 지시를 내렸으나, 신규 `mwagent.jar`로 교체되지 않고 기존 버전이 유지되는 문제.

---

## 2. 발생 원인 및 1차 분석
### 초기 증상
사용자가 직접 관리자 권한으로 `run.agent.bat`을 실행해 본 결과, 아래와 같은 로그 발생:
```
[INFO] Restart Only
[ERROR] Failed to create task, Run as Administrator.
```

### 1차 원인 (설계상 이슈)
기존 `run.agent.bat`의 MWAgent 서비스 재시작 로직은 **예약 작업(`schtasks`)**을 활용하고 있었습니다.
*   **목적:** MWAgent의 자식 프로세스인 `restart.bat`이 자신의 부모(MWAgent 서비스)를 강제 종료(`sc stop`, `taskkill`)할 때 함께 종료(동반 자살)되는 것을 방지하기 위해 **독립적인 트리거(SYSTEM 계정의 예약 작업)**를 생성.
*   **문제점:** `schtasks /Create /RU SYSTEM` 옵션은 강력한 **최고 관리자(Elevated) 권한**을 요구함. 하지만 MWAgent Java 프로세스는 일반 서비스 계정이나 UAC가 상승되지 않은 `jeus` 사용자 계정으로 실행되고 있었기 때문에 터미널 생성 시 **액세스 거부(Access Denied)**가 발생.

---

## 3. 디버깅 과정 (단계별 쪼개기 및 로깅)
결과 확인을 위해 `run.agent.bat`을 5개의 명확한 `[STEP]`으로 수정하고, 권한 상태 및 `ERRORLEVEL`을 직관적으로 출력하도록 진단용 코드를 우선 삽입했습니다.

### 3-1. 첫 번째 디버깅 (날짜 형식 오류 발견)
디버깅 스크립트를 배치 & 실행한 결과, 권한 외의 또 다른 버그를 먼저 캐치했습니다.
```cmd
[STEP 4] Create scheduled task as SYSTEM...
 Command: schtasks /Create /TN "MWAgentRestart" /TR "C:\Users\jeus\MWAgent\restart.bat" \SC ONCE /ST 00:00 /SD 01/01/2000 /RU SYSTEM /F
오류: 시작 날짜가 잘못되었습니다(올바른 날짜 형식은 "yyyy/mm/dd" 입니다).
ERRORLEVEL = -2147467269
[FAIL] Task creation failed!
```
**해결:** 한국어 Windows 환경에서는 `MM/DD/YYYY`(예: `01/01/2000`) 형식을 인식하지 못하므로, 로컬라이징에 맞춰 **`2000/01/01` (yyyy/mm/dd)**로 수정.

### 3-2. 두 번째 디버깅 (최종 권한 부족 판정)
날짜 형식을 수정한 뒤 확인된 최종 오류는 다음과 같았습니다.
```cmd
경고: /ST가 현재시간보다 빠르기 때문에 작업이 실행되지 않을 수 있습니다.
오류: 액세스가 거부되었습니다.
ERRORLEVEL = 1
```

이와 함께 스크립트 최상단에서 수집한 실행 환경(`whoami` 및 `net session` 테스트)의 결과:
*   `[OK] jeus is in Administrators group`
*   `[WARN] NOT running with elevated privileges`

**분석:** `jeus` 계정이 관리자 그룹 소속이긴 하나, 백그라운드 프로세스로 실행되었기에 **UAC(사용자 계정 컨트롤)** 권한 상승을 받지 못해 `schtasks /RU SYSTEM`을 실행할 권한이 없었습니다.

---

## 4. 해결(Workaround) 적용
`schtasks`의 SYSTEM 예약 등록은 UAC 미상승 상태에선 원천적으로 불가하므로, 다른 방식으로 **독립 프로세스 생성(Detached spawn)**을 지원하는 도구를 사용하기로 결정했습니다.

### 4-1. `run.agent.bat` : WMIC로 전환
독립된 자식 프로세스를 생성하여 부모가 죽어도 생존하는(`survival`) 특성을 지닌 `WMIC`을 활용하였습니다.
```batch
:: Spawn restart.bat as an independent process using WMIC
wmic process call create "cmd.exe /c %~dp0restart.bat" > nul 2>&1
```

### 4-2. `restart.bat` : UAC Auto-Elevation 적용
WMIC을 통해 띄워진 `restart.bat`도 부모 프로세스(`run.agent.bat`)와 동일한 "UAC 미상승" 권한을 상속받습니다. 이 때문에 `sc stop` 및 `net start MWAgent` 호출 시 `[SC] OpenService 실패 5: 액세스가 거부되었습니다.`가 발생했습니다.

이를 타개하기 위해, `restart.bat` 스크립트 실행 시작 시점에 **자신의 권한 상승(UAC 프롬프트 요청)을 자발적으로 체크 및 유도**하는 VBScript 로직(`Shell.Application runas`)을 최상단에 추가했습니다.

```batch
:: Auto-elevation to Administrator
net session >nul 2>&1
if %ERRORLEVEL% NEQ 0 (
    echo [INFO] Requesting administrative privileges...
    echo Set UAC = CreateObject^("Shell.Application"^) > "%temp%\getadmin.vbs"
    echo UAC.ShellExecute "%~s0", "", "", "runas", 1 >> "%temp%\getadmin.vbs"
    "%temp%\getadmin.vbs"
    del "%temp%\getadmin.vbs"
    exit /b
)
```

---

## 5. 최종 결과
1. `run.agent.bat`은 일반 권한에서도 오류 없이 WMIC를 사용해 백그라운드에서 `restart.bat`을 호스팅.
2. 팝업 또는 UAC 자동 상승 래퍼에 넘겨진 `restart.bat`이 관리자 권한을 획득함.
3. Windows 서비스 통제(`sc stop`, `taskkill`) 후 신규 전송받은 JAR 교체 성사.
4. MWAgent 서비스 정상 재시작 및 업데이트 적용 성공.
